# Phase 2.7: Capture Aperture & Onboarding Repair

Status: planned
Depends on: 2.6
Roadmap reference: `docs/roadmap.md` — "Phase 1: Email Identity And Inbound Capture" (claim flow) and the AR-9 consent-boundary ruling in `docs/phases/README.md`.

## Goal

The two consent-boundary capture mechanics that need no new infrastructure actually work, and the organic onboarding loop is no longer a dead end. After this phase: an unknown sender who emails `agent@keeps.email` receives the activation email with a sign-up link (today it is built but never sent — `email.sender_unknown` has no consumer); auto-responder mail loops are structurally impossible (RFC 3834 guards); a counterparty's reply on a thread the user already shared attaches to that user's thread instead of dying in `pending_inbound_emails` (CC-once → whole thread followed); and a capture-recipes doc tells design partners how to scope auto-forwarding they own (Gmail filters). The deferred Phase 2.6 claim-path smoke passes end to end.

## Why Now

Design partners cannot onboard organically until the activation email sends — it is literally the top of the funnel, and the gap was found (2026-06-12) only because the claim-path smoke was deferred. Auto-replying to strangers without loop guards risks reputation-damaging mail storms (their out-of-office answers our auto-reply, forever), so the guards must land in the same phase, before the first stranger emails the agent. Thread-following is the highest-leverage answer to the receive-time capture problem (AR-9): it converts one explicit act (CC the agent) into ongoing coverage using nothing but email's native mechanics — the thread-key machinery already exists from Phase 1. All of this must precede design partners; none of it depends on Phase 3.

## Deliverables

1. **Auto-reply detection on normalized inbound email.**
   Acceptance: `src/email/normalize.ts` captures `Auto-Submitted`, `Precedence`, and `X-Auto-Response-Suppress` headers into `NormalizedEmail.headers`. A pure helper `isAutoGenerated(email: NormalizedEmail): boolean` in `src/email/auto-reply.ts` returns true when `Auto-Submitted` is present and not `no`, or `Precedence` is `bulk | junk | list | auto_reply`, or the From address's domain is one of our own sending/reply domains (derived from `POSTMARK_FROM_ADDRESS` and `POSTMARK_REPLY_TO_BASE` — never hardcoded). Unit tests cover Gmail vacation responder, Outlook OOO (`X-Auto-Response-Suppress`), mailing-list `Precedence: list`, and a self-loop fixture From `agent@keeps.email`.

2. **Activation email actually sends, loop-safely (consumer for `email.sender_unknown`).**
   Acceptance: `src/workflows/functions/send-activation-email.ts` registers `send-activation-email` triggered by `email.sender_unknown`, with `idempotency: "event.data.pendingInboundEmailId"`. Steps: (a) load the pending row + its raw payload; re-derive `isAutoGenerated` and **skip** (audit `email.activation_suppressed`, reason in metadata) when true; (b) skip when an activation email was already sent to this sender within the pending-retention window (7d) — checked via `pending_inbound_emails.activation_sent_at` for any row with the same `sender_email`; (c) otherwise send `buildUnknownSenderReply` via `EmailSender` equivalent for non-nudge mail (a small `sendSystemEmail` helper in `src/email/system-send.ts` that sets `Auto-Submitted: auto-replied` on OUR outgoing auto-reply, per RFC 3834, and persists an `outbound_emails` row with `nudge_id` null), stamp `activation_sent_at`, audit `email.activation_sent`. The function never replies to its own suppression cases — there is no path that answers an auto-generated email with mail.

3. **CC-once thread following: attach known-thread counterparty replies.**
   Acceptance: in `handlePostmarkInboundEmail`, before the unknown-sender branch, a thread-follow check runs: compute `buildThreadKey(normalized)`; if an `email_threads` row with that key exists for some user AND the sender appears in that thread's prior participants (any prior message's From/To/Cc on that thread — comparison via `normalizeIdentityEmail`), persist the email as a regular `inbound_emails` row for that user (existing `createInboundEmailForUser` path), emit `email.received`, audit `email.thread_followed`, and return `sender_verified`-shaped 202. Unknown senders with no matching thread (or matching thread but NOT a prior participant — spoof guard) fall through to the pending path exactly as today. Repository gains `findThreadOwnerForFollow(threadKey, senderEmail)` with unit tests covering: counterparty reply attaches; stranger with stolen `References` does NOT attach; user's own new thread unaffected.

4. **Capture recipes doc + activation copy.**
   Acceptance: `docs/recipes/capture.md` documents, in design-partner-facing language: (a) BCC/forward at write time; (b) CC-once thread following ("CC agent@keeps.email once; the whole thread is covered" — shipped by deliverable 3); (c) the user-owned Gmail filter recipe (filter → auto-forward to the agent address; notes that Gmail's forwarding-confirmation email arrives at our inbound webhook and how to retrieve the confirmation link from the pending row until self-serve surfacing exists); (d) one paragraph on the consent boundary (AR-9) for trust-sensitive workplaces. The activation email copy in `buildUnknownSenderReply` is reviewed against this doc's tone (calm, no marketing).

5. **Onboarding stepper: inline custom sign-up, one thing at a time, mobile-first (added 2026-06-12).**
   The stepper (`app/get-started-stepper.tsx`) already has the design language (warm cream/terracotta, Tailwind 4). What changes:
   - **Inline Clerk sign-up via hooks, not the prebuilt form.** The email step no longer redirects to `/sign-up` (Clerk's styled `<SignUp/>` — a jarring style break). Instead the stepper drives Clerk headlessly with `useSignUp`: email submit → `signUp.create({ emailAddress })` + `prepareEmailAddressVerification({ strategy: 'email_code' })` → a new in-stepper **code step** (single input, auto-advance on 6 digits, resend link with cooldown) → `attemptEmailAddressVerification` → `setActive(...)` → continue to the capture step in place. Error states render inline (invalid email, wrong code, rate-limited, email already exists → link to `/sign-in`). The existing `/sign-up/[[...rest]]` route stays as a fallback (deep links from activation emails land there) but the happy path never leaves `/`. `/sign-in` keeps Clerk's prebuilt component — not worth custom work at pilot traffic.
   - **Fix the capture address.** `captureAddress` is hardcoded `agent@keeps.ai` — a domain we do not own. Replace with a single exported constant in `src/product/capture-address.ts` reading `NEXT_PUBLIC_CAPTURE_ADDRESS` (default `agent@keeps.email`); stepper, activation email copy, and recipes doc all derive from it.
   - **Mobile-first audit.** Every step usable one-handed on a 390px viewport: inputs ≥16px font (no iOS zoom), tap targets ≥44px, the code input uses `inputmode="numeric" autocomplete="one-time-code"` (iOS SMS/mail code autofill), no horizontal scroll, stepper progress visible without scrolling. Verified at 390×844 and 430×932 via Playwright or manual device pass.
   Acceptance: a fresh user on a phone completes email → code → capture instructions → working style without ever seeing a Clerk-styled page; the displayed capture address is `agent@keeps.email`; sign-up still creates the Clerk user, fires the `user.created` webhook, and claims held emails exactly as the 2.6 flow did.

6. **keeps.email is the canonical web origin.**
   Acceptance: the `keeps.email` domain is attached to the Vercel project (done 2026-06-12 — already serving). `NEXT_PUBLIC_APP_URL` flips to `https://keeps.email` and prod redeploys; activation-email links and the stepper's post-signup URLs use it. `keeps-ivory.vercel.app` remains as a secondary alias (Inngest `INNGEST_SERVE_ORIGIN` may stay on it — it is stable — or move; either way the Inngest sync must be re-verified after the env flip). Clerk dev-instance origin allows `https://keeps.email` (verify sign-up works from the new origin; dashboard change is Arav's if needed).

## Risks & Open Questions

1. **Participant extraction for the spoof guard.** Prior participants come from stored thread messages' address fields. **Default:** match against `inbound_emails.raw_payload` To/Cc/From of the thread's existing rows; good enough for the pilot. Revisit if threads grow long enough to make the scan slow (they won't at pilot scale).
2. **Activation rate limit scope.** Per-sender 7d window (matches pending retention). **Default:** keep; one activation email per stranger per week is the polite ceiling regardless of how many emails they send.
3. **Gmail forwarding-confirmation surfacing.** V1 is "we fish the link out of the pending row when onboarding a design partner by hand." Self-serve surfacing is deliberately deferred (Out of Scope) — it needs a settings surface that doesn't exist yet.

## Out of Scope

- Any inbox OAuth or sender-level watching that requires reading unconsented mail (AR-9 — permanently out, not deferred).
- Self-serve surfacing of Gmail forwarding confirmations (needs settings UI; Phase 5+ at earliest).
- Digest coverage line and reply-to-digest capture prompt (Phase 3, deliverable 8 — specified there).
- Calendar-triggered capture suggestions (Phase 4; AR-9-compliant via metadata only).
