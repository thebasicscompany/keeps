## Why this sprint exists (the gap, diagnosed 2026-06-13)

Phases 0–6 are DONE + live on keeps.email. Capture → extract → nudge → recall all work. But the product is functionally a per-email **loop extractor**, not a context **engine**. Three structural gaps:

1. **Extraction is stateless.** `extractLoops({ email })` (src/agent/extract-loops.ts) prompts the model with only `Subject / From / Participants / body` — never prior loops, thread history, known people, or what you've committed before. The 5th email about a deal extracts as if the first 4 never happened.
2. **No graph.** `loops.ownerText` / `requesterText` are free-text columns (src/db/schema.ts ~425). There is NO people / companies / entities table. The vision's "promote entities to first-class rows by Phase 7 at latest" (project_keeps_vision.md / memory `keeps-vision`) is undone. "Show Acme loops" is a string match over a flat list.
3. **No reconciliation or synthesis.** New emails (incl. a counterparty "done!" reply on a CC'd thread) spawn NEW loops; nothing finds/closes/advances the matching open loop. "Insights" is deterministic bucketing, not understanding.

Net: CC'ing more email gives you MORE loops, not a SMARTER assistant. This sprint builds the substrate: an entity graph, context-aware extraction that reconciles, and synthesis-grade recall.

Ground truth: `docs/roadmap.md`, `docs/phases/README.md` (AR-1..AR-9 — esp. **AR-8 model boundary** and **AR-9 consent boundary + suggestion provenance**), memory `keeps-vision` + `keeps-status`, and git log. The naming: call this **Phase 7 "Context Engine"** (it absorbs the old roadmap Phase-7 "entity promotion" option-keeper and precedes team/MCP; confirm numbering with Arav). The Phase 5 + Phase 6 sprint handoffs in this dir show the exact orchestration mechanics that worked.
