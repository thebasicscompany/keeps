## Why this sprint exists (the gap, diagnosed 2026-06-13)

Phases 0–6 are DONE + live on keeps.email. Capture → extract → nudge → recall all work. But the product is functionally a per-email **loop extractor**, not a context **engine**. Three structural gaps:

1. **Extraction is stateless.** `extractLoops({ email })` (src/agent/extract-loops.ts) prompts the model with only `Subject / From / Participants / body` — never prior loops, thread history, known people, or what you've committed before. The 5th email about a deal extracts as if the first 4 never happened.
2. **No graph.** `loops.ownerText` / `requesterText` are free-text columns (src/db/schema.ts ~425). There is NO people / companies / entities table. The vision's "promote entities to first-class rows by Phase 7 at latest" (project_keeps_vision.md / memory `keeps-vision`) is undone. "Show Acme loops" is a string match over a flat list.
3. **No reconciliation or synthesis.** New emails (incl. a counterparty "done!" reply on a CC'd thread) spawn NEW loops; nothing finds/closes/advances the matching open loop. "Insights" is deterministic bucketing, not understanding.

Net: CC'ing more email gives you MORE loops, not a SMARTER assistant. This sprint builds the substrate: an entity graph, context-aware extraction that reconciles, and synthesis-grade recall.

## Design validation (added 2026-06-13, after a research pass on the hard sub-problems)

Before executing, the four hard sub-problems were validated against 2024–2026 best practice (record linkage, LLM entity-matching, RAG context scoping, agent-memory systems). **The architecture below independently matches the consensus** — the unifying principle everyone converges on: *the LLM does fuzzy semantic judgment; deterministic code holds the merge pen and decides what is true.* Key framing to keep in mind throughout: **a false merge is a PRECISION failure; a duplicate loop is a RECALL failure — and they are asymmetric.** A duplicate is visible and cleanable; a false merge is silent and destructive. So every merge-class decider is tuned for precision, accepting lower recall. A corollary that simplifies the whole design: **the candidate-retrieval stage can never cause a false merge — only the final decision can** — so retrieval (context scoping) can be cheap/loose, and the decision must be strict.

Four research-driven upgrades over the original plan (folded into the corrections + waves below):
1. **Three-band reconciliation, not binary** (Fellegi-Sunter / every production ER system): auto-reconcile (high) / create-new (low) / **ask the user (uncertain middle)**. Fits Keeps' explicit-capture, human-in-the-loop posture — see correction #4.
2. **Select-among-candidates prompt framing**, not per-loop yes/no (ComEM, COLING 2025: comparing across candidates beats isolated matching) — see B1.
3. **Anti-sycophancy prompt design**: injecting candidate loops primes a false "yes, match"; require the model to cite the shared identifier justifying any link, and make create-new the explicit safe default — see B1.
4. **Auto-CLOSE gated as strictly as auto-MERGE** (closing a still-open commitment is its own silent trust failure) — see B2.

**On Mem0 / Graphiti / memory frameworks (build vs buy — decided 2026-06-13/14):** Separate TWO things called "memory". (1) The **structured commitment graph** (loops + entities + loop_events) is the CORE product — it must be relational, owned, auditable (loop_events / AR-9), deterministic-fallback-capable, and routed through mutateLoopState (AR-8). Build it NATIVELY; do NOT hand it to Mem0 or Graphiti. Postgres already gives us a temporal property graph (loops + loop_events bi-temporal-ish + entities + loop_entities), live in prod with ~1300 tests; adopting a memory framework means a second datastore (Mem0=vector store; Graphiti=Neo4j/FalkorDB) dual-writing the one thing that must never be wrong, AND it takes the merge pen out of our hands (both do entity/edge dedup with their own internal LLM calls + thresholds — inverting the three-band/ask, conservative, no-creds-fallback decider that is the whole point). The hard part of this sprint is the reconciliation DECIDER, not "build a graph" (the entities table is one migration). So: build native. Borrow the pattern from Mem0 (extract → retrieve top-k → action-verb ADD/UPDATE/NOOP, two-pass, small candidate set, ADD-only safe default). (2) A **soft semantic-memory** layer for non-loop context ("Sarah prefers Tuesday calls", "Acme FY ends in March") feeding INTO extraction is a genuine v2+ opportunity — and there **Graphiti > Mem0** (temporal graph fits "context that evolves" better than flat memories). Evaluate Graphiti-on-FalkorDB vs extending Postgres THEN, as a layer feeding the decider — never as owner of the commitment graph or the merge decision. Not v1 scope.

**Tool survey (done 2026-06-14; ~15 tools across agent-memory + entity-resolution/dedup):** confirmed build-native for v1. Two disqualifying facts: (a) **No credible TS/Node entity-resolution library exists** — every serious ER engine (Splink, Zingg, dedupe, RecordLinkage, Senzing) is Python/JVM = a second runtime/deploy on our Vercel/Inngest/Node stack. (b) **Graphiti is Python-only and forces Neo4j/FalkorDB** (no Postgres backend — open issue #779) and auto-merges nodes/edges via internal LLM calls = the false-merge machinery we forbid, non-deterministic + unauditable. Deeper reason the whole category misses: these tools optimize to AUTO-MERGE at scale (accepting a false-merge rate for throughput); we want ABSTENTION at tiny scale with a near-perfect natural key (email). At hundreds–thousands of records the statistical machinery (EM/blocking) gets *noisier*, not better. **"Native" ≠ "from scratch":** we BUY the hard algorithmic parts as Postgres extensions and copy proven schemas — (i) **`pg_trgm` + `fuzzystrmatch`** (trigram similarity, Levenshtein, Metaphone) drive name-similarity SUGGESTIONS into a review queue, never auto-merge; optional pure-TS `talisman`/`natural` (Jaro-Winkler/Metaphone) if in-app scoring is wanted; (ii) copied data models — Mem0 verb taxonomy (ADD/UPDATE/DELETE/NOOP → create/advance/close/noop), Graphiti bi-temporal schema (valid-time vs ingestion-time columns + invalidate-don't-delete), Splink Fellegi-Sunter three-band (already our decider), OpenRefine human-confirm UX (the "ask" band). We write only the ~few-hundred-line conservative resolver + reconciliation decider — the part that must be ours because the false-merge guarantee lives there. **v2 semantic recall** ("where's the Acme deal?", read-only, no merge risk): `pgvector` in the existing RDS, or LlamaIndex.TS `VectorMemoryBlock` (TS-native, MIT) scoped to recall-only. Not v1.
